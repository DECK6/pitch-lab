# Plan 02-01 — Product P1 practice and harmony

## Objective

Ship a regression-safe TUNING/PRACTICE split. PRACTICE must explain a selected key through diatonic, color, and related chords; play those choices at useful loudness; and grade a singer against a chosen chord tone. The existing V1 tuner and optional Neural path remain intact.

## Architecture

```text
                         ┌─ TuningWorkspace ─► existing tuner UI
AppShell ─► ModeStore ───┤
                         └─ import(PracticeWorkspace)
                                      │
KeyContext ─► HarmonyEngine ─► ChordCatalog ─► chord lanes
                                      │
                                      ├─► ReferenceVoiceBank
                                      └─► TargetExercise
AudioSession ─► PitchFrame ───────────────► TargetComparator
```

`AudioSession` belongs to `AppShell`, not either workspace. This is the structural change that makes mode switching safe and gives P2 a future SCORE workspace without adding a second microphone lifecycle.

## Test coverage and failure paths

| Path | Failure | Automated evidence | User-visible recovery |
|---|---|---|---|
| Mode import | Practice chunk fails | E2E route failure | Stay in TUNING; show retry |
| Stored mode/key | Corrupt local value | Unit + E2E | Reset to TUNING / C major |
| Theory catalog | Wrong enharmonic spelling | 24-context golden suite | Build fails before release |
| Chord audition | Voice leak or clipping | Offline audio + lifecycle tests | Stop all voices; retain controls |
| Mic grading | Reference sound grades itself | Gate integration test | Evaluation pauses during playback |
| Target scoring | Stale/unvoiced frames pass | Reducer boundary tests | WAITING FOR STABLE PITCH |
| Mobile gestures | Lane blocks page scroll | Touch E2E | Independent horizontal lane; vertical page remains available |
| Bundle | Practice enters boot graph | Build graph + request E2E | Build fails, never deploys |

No P1 path may fail silently while displaying a stale pass result.

## Implementation order

Wave 1 is structural and lands alone. Wave 2 establishes pure theory contracts. Wave 3 can split into two workstreams after Wave 2: the practice UI and audio voice bank touch different primary modules, though both must coordinate through stable chord IDs. Wave 4 consumes both. Wave 5 closes release gates.

```text
Wave 1: App shell/modes
          │
Wave 2: Theory engine
          │
          ├───────────────┐
Wave 3A: Practice UI   Wave 3B: Voice bank
          └───────┬───────┘
Wave 4: Target exercise/scoring
          │
Wave 5: Bundle, browser, physical-device release gates
```

## NOT in scope

- Score import, score rendering, rhythm timeline, PDF OMR, or choir extraction.
- Polyphonic microphone recognition.
- Advanced jazz reharmonization or user-authored chord rules.
- Sample libraries, MIDI, accounts, or cloud persistence.
