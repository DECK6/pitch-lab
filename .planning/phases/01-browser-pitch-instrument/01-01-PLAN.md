# Plan 01-01 — Working browser pitch instrument

## What already exists

- Greenfield codebase: no application code to preserve or duplicate.
- Reuse Pitchy/McLeod for Light detection, Web Audio for capture/reference tones, SwiftF0 for neural F0, and ONNX Runtime Web for WASM inference.
- Reuse the approved wireframe and product spec rather than creating a second visual or architecture concept.

## Architecture

```text
Mic MediaStream
      │
AudioContext ──► AudioWorklet collector ──► EngineManager
                                              │
                              ┌───────────────┴───────────────┐
                              ▼                               ▼
                      Light Worker                    Neural Worker
                    Pitchy / MPM           resample 16 kHz → SwiftF0/ORT
                              │                               │
                              └───────────────┬───────────────┘
                                              ▼
                                     normalized PitchFrame
                                              ▼
                              smoothing → note/cents/trail UI

Reference keyboard ──► OscillatorNode ──► detector gate
```

Neural selection is a transaction: keep Light live, fetch/initialize/warm Neural, switch at a chunk boundary only after success, and dispose/fall back without touching the microphone session on failure.

## Failure modes

| Path | Real failure | Test | Handling | User sees |
|---|---|---|---|---|
| Permission | Denied or prompt never resolves | E2E/manual | cancel/retry state | Actionable permission message |
| Capture | AudioContext suspends or track ends | Unit/E2E | one automatic resume, then tap-to-resume/restart | Explicit interruption state |
| Light | Worker crashes or queue drops | Integration | restart/fresh discontinuity | Brief recovering state, no stale note |
| Neural fetch | Offline, wrong MIME, abort, checksum mismatch | Unit/E2E | abort/dispose and retain Light | Failure reason and retry |
| Neural inference | Tensor drift or mobile backlog | Integration/unit | reject adapter or performance fallback | Device-suited-to-Light message |
| Piano | Pointer release is missed | Unit/E2E | pointer capture plus timed release cleanup | No stuck tone |
| Build | ORT emits extra backend or exceeds cap | Build test | fail production verification | Build failure, never silent |

No path is allowed to fail silently while retaining a stale note.

## Performance controls

- One in-flight and one latest-pending PCM message per worker; no growing queue.
- Transferable buffers are reused; UI history is time-bounded.
- Neural assets are dynamically imported and production-budgeted separately.
- Neural worker uses single-thread WASM on iOS and falls back when rolling p95 exceeds the mobile budget.
- Main thread renders on animation frames and coalesces audio updates.

## Dependencies

- Phase is sequential because capture, both engines, and UI share the `PitchFrame` and session contracts.
- Model source and checksum are pinned in `docs/product-spec.md`.
- Physical iPhone latency/memory verification cannot be simulated reliably; automated checks produce a release candidate and an explicit manual gate.

## NOT in scope

- PDF score import, grading, and choir separation: a later product milestone.
- Polyphonic/source-separated audio: a different signal-processing problem.
- Backend services, authentication, analytics, or PCM persistence: not needed and conflict with privacy/weight.
- Sampled piano, sustain, MIDI, and polyphony: do not improve the v1 reference-note job enough to justify their weight.

## Engineering Review Summary

- Step 0 scope challenge: accepted as-is. The file count is justified by hard browser isolation boundaries (worklet, Light worker, Neural worker, build scripts, tests), while runtime dependencies stay at two.
- Architecture: no open design issue; failure isolation and atomic engine switch are explicit.
- Code quality: explicit state/contracts and pure DSP helpers prevent UI coupling and stale-value bugs.
- Tests: every planned branch maps to unit, worker, build, or browser evidence; physical microphone/device checks are marked manual rather than faked.
- Performance: queue bounds, reuse, lazy loading, single-thread iOS WASM, and hard build budgets address the primary risks.
- Parallelization: sequential implementation, no safe worktree split because all streams share the initial audio and frame contracts.
- Lake score: 5/5 complete recommendations selected.
