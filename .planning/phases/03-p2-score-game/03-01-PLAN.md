# Plan 03-01 — Product P2 score game and choir parts

## Objective

Turn MusicXML/MXL or corrected PDF OMR output into one confirmed monophonic target line. Render the score, follow its key and timing with an audio-clock-driven game, and let choral singers choose or repair S/A/T/B mappings before grading.

## V2 preview execution update — 2026-07-22

The current branch implements the bounded browser preview before the full roadmap below:

- [x] renderer-independent score contracts and SATB fixtures;
- [x] bounded local MusicXML/MXL parsing, ties/transpose/key/tempo/meter, and unsafe XML/archive rejection;
- [x] explicit/condensed choir-line candidates with confidence and manual role/octave/note correction;
- [x] count-in, tempo scaling, pause/resume, looping, note grading, and cursor driven by an injected `AudioContext.currentTime` clock;
- [x] local PDF.js rendering plus experimental staff/barline/notehead detection with a mandatory correction gate;
- [x] separate initial, practice, SCORE, PDF OMR, and Neural bundle budgets and on-demand browser request tests.

This preview does not claim the later full-engraving renderer, workerized long-score parser, repeat/ending expansion, metronome/reference melody, production OMR corpus thresholds, or physical-device certification. Repeats/endings and jump navigation are blocking review warnings. The full tasks below remain the roadmap for those gates.

## Architecture

```text
File ─► Import adapter ─► MusicXML validator ─► ScoreNormalizer
                                                  │
                           ┌──────────────────────┼──────────────────────┐
                           ▼                      ▼                      ▼
                    PartExtractor          ScoreRenderer          TimelineEngine
                           │                      │                      │
                           └──────── selected VoiceLine ────────────────┤
                                                                        ▼
AudioContext clock ─► LookaheadScheduler ─► target windows + score cursor
AudioSession ─► PitchFrame ─► LatencyCompensator ─► NoteScorer
```

The parser owns musical truth. The renderer owns pixels. The game owns playback order and time. The scorer consumes one confirmed `VoiceLine`. Keeping these boundaries explicit prevents an OSMD upgrade or an OMR error from changing scoring semantics invisibly.

## Test and failure matrix

| Path | Real failure | Required test | Handling | User sees |
|---|---|---|---|---|
| MXL decode | Zip traversal/bomb or missing root | Unit/security fixture | Reject before XML parse | Exact invalid-file reason |
| XML parse | Malformed or external entity | Unit/security fixture | Safe parser rejection | Replace-file action |
| Normalize | Unknown construct or ambiguous jump | Golden fixture | Preserve warning/ungradable span | Measure-linked warning |
| Renderer | Long score exhausts mobile memory | Perf E2E | Incremental/windowed render, dispose old pages | Progress or reduced preview, not freeze |
| Choir mapping | Condensed/crossing voices | Fixture + E2E | Confidence + manual mapping | Cannot start until confirmed |
| Scheduler | rAF stalls/background tab | Fake clock + E2E | Audio clock pause/resume | Session paused, no unfair misses |
| Scoring | Input latency shifts onsets | Synthetic + device | Stored calibration offset | Calibrate/retry message |
| PDF upload | Offline/timeout/process crash | Service integration | Cancel, cleanup, retry | Job status and retry |
| OMR semantics | Accidental/tie misread | Corpus + correction E2E | Block grading until confirmed | Suspicious measures highlighted |
| Privacy cleanup | Temp file survives failure | Integration/restart test | Sweeper + TTL + audit evidence | No silent enablement; feature gate stays off |
| Bundle | Renderer enters initial graph | Build/request test | Fail build | Never deployed |

Any path with unknown musical meaning must become a warning or ungradable span. It must never become a guessed target note.

## Workstream order

```text
Wave 1: Contracts + renderer spike
              │
Wave 2: Parser/normalizer
              │
              ├──────────────────┐
Wave 3A: Renderer/UI       Wave 3B: Choir extraction
              └──────────┬───────┘
Wave 4: Scheduler + scoring
              │
Wave 5: OMR service + correction
              │
Wave 6: Corpus, privacy, bundle, physical-device release gates
```

Renderer/UI and choir extraction may run in parallel after the score contract stabilizes. OMR starts only after the same parser and correction contracts work on local MusicXML; otherwise OMR bugs and parser bugs become indistinguishable.

## NOT in scope

- Handwritten OMR or arbitrary non-Western notation recognition.
- Full engraving editor or automatic layout repair.
- Polyphonic microphone grading or accompaniment/source separation.
- Automatic SATB identity based solely on pitch range.
- Accounts, cloud audio, social scores, or a silent production cutover.
